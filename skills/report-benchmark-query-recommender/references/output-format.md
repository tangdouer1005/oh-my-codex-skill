# Output Format

Use this structure when recommending benchmark queries from a report's third section.

This exact structure should be used both in the assistant response and in the Markdown artifact saved to disk.

At the top of the saved Markdown, add:

- `Source file`: absolute path to the input summary Markdown
- `Generated by`: `report-benchmark-query-recommender`
- `Output time`: timestamp if available in the current environment

## 1. Section-3 Assessment

Write 3 short bullets:

- `Entity level`: what one row-like unit represents
- `Available ingredients`: which tables or field groups are available
- `Reasoning opportunities`: which cross-source joins or derived metrics are feasible

## 2. Recommended Queries

Recommend at least 10 candidates unless the user asks for a different number.

Across the full set, maintain diversity in reasoning pattern, entity scope, chart type, and metric family. Avoid near-duplicate queries.

For each candidate, use this template:

### Query N
- `Query`: a complete natural-language chart instruction that explicitly includes all of the following when relevant:
  - chart type
  - entity scope
  - metric definition
  - time window
  - unit
  - sorting rule
  - top-`N` rule or final selection rule
  - filter or inclusion rule
  - whether the chart shows absolute values, percentages, shares, growth rates, or ratios
- `Why this fits the benchmark`: one short paragraph focused on why this is not a direct-plot task
- `Required sources`: explicit section-3 tables, fields, page spans, or entity blocks that must be combined
- `Derived data needed`: the intermediate calculations or joins needed before plotting
- `Reasoning chain`: describe the intended pipeline in 4 to 6 short steps, ideally including filtering, cross-block retrieval, derivation, ranking, subset selection, and charting
- `Chart spec check`: confirm in one short line that the `Query` already states chart type, unit, and comparison rules explicitly
- `Difficulty`: `medium` or `hard`
- `Risk note`: the main ambiguity or annotation risk

## 3. Excluded Query Types

List 2 to 4 tempting but weak candidates that should be excluded, with one-line reasons such as:

- direct transcription
- ambiguous denominator
- unsupported by the available fields
- too speculative

## 4. Diversity Check

After the recommendations, add 3 to 5 short bullets covering:

- which reasoning patterns are represented
- which chart types are represented
- which entity scopes are represented
- which kinds of redundancy were intentionally avoided
- whether any diversity dimension is still weak because of source limitations

## Writing Style

- Keep the query itself user-facing, but do not trade away precision for brevity.
- Prefer one fully specified sentence over a short but underspecified sentence.
- Keep rationale paragraphs short but specific.
- Prefer concrete field names from the report summary over generic words like "some metrics."
- If the report uses English field labels, keep those labels in English.
- If the query involves money, explicitly specify whether the output should use `$B`, `$M`, or the original report units.
- If the query involves a derived metric, name the formula in natural language inside the query.
- If the intended benchmark difficulty depends on ranking or top-`N` selection, write that step explicitly in the query instead of leaving it implicit.
- When writing a batch of queries, vary sentence openings and query structure slightly so the set does not read like a templated clone.
