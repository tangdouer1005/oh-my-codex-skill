# 飞书周报维护

这个 skill 用于持续维护一份飞书周报，而不是只生成一次成稿。

周报由两层组成：

- 上半部分是 `本周整理`，按项目归纳
- 下半部分是 `本周 Log`，尽量保留用户原始表述

当用户会在一周内反复补充进展，希望 Codex 持续把这些内容写入同一份飞书文档时，就应该使用这个 skill。

## 适用场景

- 用户说“帮我维护周报”
- 用户说“把这条记到本周周报里”
- 用户希望在飞书里持续维护每周记录
- 用户要求按项目整理周报，而不是按“已完成 / 进行中 / 下周计划”拆分
- 用户希望下半部分保留原始 log，方便回溯

如果用户只是要整理会议纪要，不要用这个 skill，优先使用 `lark-workflow-meeting-summary`。

## 默认文档结构

除非用户明确要求其他结构，否则默认采用下面这套飞书 Wiki 组织方式：

- 根节点：`每周记录`
- 每周子文档标题：`<开始日期>-<结束日期>`
- 例如：`2026-04-20-2026-04-26`

默认以周一作为每周起始日。如果用户明确要求别的周定义，再按用户要求处理。

## 周报正文结构

周报文档里只保留内容本身，不写维护规则或使用说明。

使用下面的结构：

```markdown
时间范围: <开始日期> 至 <结束日期>

## 本周整理

### 项目：<项目标题>

- 背景：<项目背景>
- 本周进展：<本周发生了什么>
- 当前结论：<当前状态、结果或判断>
- 后续动作：<下一步动作，如已知>

### 项目：<项目标题>

- 本周进展：...
- 当前结论：...
- 后续动作：...

## 本周 Log

### <yyyy-mm-dd>

- <原始记录 1>
- <原始记录 2>
```

说明：

- `本周整理` 是给人读的周报摘要
- `本周 Log` 是过程记录和证据
- 不要在正文里加入“维护约定”“使用方法”“规则说明”等内容

## 维护规则

### 1. Log 要尽量保留原始语意

当用户发来新的进展时：

- 先追加到当周的 `本周 Log`
- 尽量保留用户原话
- 只做必要的轻量整理：
  - 句子过于省略时，补足最少量主语让它能读懂
  - 一句话里混了两件事时，可拆成两条
  - 日期表达明显不统一时，可规范成统一日期形式

不要把 Log 写得太“漂亮”。Log 的目的就是保留原始过程。

### 2. 上半部分必须按项目整理

`本周整理` 必须按项目来组织，项目名可以对应一个明确的工作主题、任务线或持续事项，但顶层结构必须是“项目”，不能按泛化状态桶组织。

好的项目标题示例：

- `项目：DataChef 论文复现`
- `项目：保研面试安排流程`
- `项目：中期答辩材料整理`

不推荐作为本 skill 顶层结构的标题：

- `已完成`
- `进行中`
- `下周计划`

如果一条新 Log 属于已有项目，就合并进对应项目；如果它开启了新的明确工作线，再新建项目。

### 3. 项目背景是必填项

每个项目都必须有 `背景`。

规则如下：

- 第一次出现某个新项目时，如果现有上下文不足以写出清楚背景，要向用户补充确认
- 一旦这个项目已经建立过背景，后续周报优先沿用旧背景，不要每周重新向用户索要
- 优先从上一周周报中读取该项目的背景描述，再按本周实际情况做最小必要更新
- 如果上一周的背景已经过时或明显失真，再向用户确认是否需要改写

### 4. 尽量保持项目标题稳定

同一周内，项目标题尽量稳定，不要因为用户换了个说法就频繁重命名。

只有在原来的标题明显不准、会误导后续维护时，才重命名该项目。

### 5. 每次更新都维护两层

对于有意义的周内更新：

1. 先把原始内容追加到 `本周 Log`
2. 再同步更新对应的 `本周整理`

如果一条信息太模糊，不足以安全归纳：

- 仍然要记到 `本周 Log`
- 上半部分可以暂时不动，或只写非常保守的概括

### 6. 整理区要短而清楚

`本周整理` 应该像一份简洁可读的周报，而不是把 Log 再抄一遍。

- 合并重复进展
- 把零碎动作压缩成更高层的一条进展
- 优先保留决策、里程碑、阻塞点、下一步
- 不重要的寒暄或细枝末节，不必写进整理区

## 工作流程

### 第 1 步：确定要维护哪一周

根据当前日期计算周范围，并在标题和 `时间范围` 中使用明确日期。

如果用户说“这周”“上周”“本周”这类相对时间，要先解析成明确日期，再创建或更新文档。

### 第 2 步：确认根节点存在

默认目标结构：

- Wiki 根节点 `每周记录`
- 下面按周放子文档

建议搭配这些能力：

- 先读 `lark-shared`，确认认证和权限处理方式
- 用 `lark-doc` 创建和更新文档
- 用 `wiki +node-create` 创建根节点
- 在权限允许时，用 `docs +search` 查找已有根节点或已有周文档

如果因为应用审批或 scope 限制无法搜索，要简短说明，然后使用当前最安全的兜底方式继续，不要一直卡住。

### 第 3 步：确认当周子文档存在

创建或复用标题为 `<开始日期>-<结束日期>` 的子文档。

如果是新建文档，初始化为默认模板，并给 `本周 Log` 先建当天的小节。

### 第 3.5 步：复用上一周的项目背景

当本周第一次写入某个项目时：

- 先检查上一周周报里是否已有同名项目
- 如果有，优先复用上一周的 `背景`
- 如果只有部分可复用信息，就沿用主体表述并做最小修改
- 如果上一周没有这个项目，或旧背景已经不足以支撑本周描述，再向用户确认背景

不要因为是新的一周，就把所有项目背景重新向用户问一遍。

### 第 4 步：追加新的 Log

新增用户进展时：

- 优先追加到今天日期对应的小节下
- 如果今天的小节不存在，就新建
- 记录内容尽量贴近用户原话

### 第 5 步：刷新项目整理区

在写入 Log 后，同步更新 `本周整理`：

- 找到最匹配的项目
- 如果是新项目，先补齐背景
- 更新 `本周进展`
- 如果用户的新信息改变了当前判断，就更新 `当前结论`
- 如果下一步更清楚了，就更新 `后续动作`

如果消息里出现阻塞点或未决问题，优先把它放在 `当前结论` 或 `后续动作` 中，而不是默认额外创建“问题”总章节，除非用户明确要求。

## 飞书操作建议

优先使用下面这些操作：

- `docs +create` 创建周报文档
- `docs +update --mode append` 追加新的 Log
- `docs +update --mode replace_range` 或 `--selection-by-title` 更新某个项目小节
- `drive files patch` 在需要时修正文档标题
- `wiki +move` 仅在需要调整到 `每周记录` 根节点下时使用

当文档已经进入持续维护阶段后，尽量避免频繁使用 `overwrite`，除非用户明确要求整体重建，或文档还停留在刚初始化模板的阶段。

## 整理区写作风格

- 具体，不空泛
- 用事实和进展说话，不要夸张拔高
- 让扫读的人也能快速看懂这一周发生了什么
- 尽量保留用户隐含的优先级
- 不要编造结论、决策或下一步

## 典型触发语句

- “帮我在飞书里维护周报，我之后会陆续告诉你进展”
- “把这条也记到本周记录里”
- “整理一下这周周报，按项目归纳，不要按完成状态写”
- “把刚才那条同步到本周的飞书周报”
