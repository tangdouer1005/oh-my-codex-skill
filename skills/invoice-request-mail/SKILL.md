# 开票申请邮件

在以下整条链路中使用本 skill：

1. 读取飞书文档或本地截图。
2. 提取可报销支付记录。
3. 核对日期、金额、数量是否和用户预期一致。
4. 按规则分组成多封开票申请邮件。
5. 在飞书邮箱中创建带附件的草稿。
6. 只有在用户明确确认后才发送。

优先保证流程可核对、可追踪。大多数情况下，附件分组的正确性比处理速度更重要。

## 先确认的输入

在创建草稿前，先确认或推断这些信息：

- 数据来源：飞书 wiki/doc 链接、本地截图，或用户给出的现有邮件模板。
- 要使用的具体章节，例如 `可以报销（api）`。
- 分组规则：
  - `-1799.xx` 这类记录默认视为一笔不可拆分的 `1800`，除非用户明确允许拆分。
  - `-899.xx` 这类记录默认视为一笔 `900`。
  - 当用户要求“尽量按 1800 开票，剩余 900 单独一封”时，再把两笔 `900` 配成一封 `1800`。
- 收件人邮箱，通常是 `receipt@chatanywhere.tech`，但仍以用户确认或模板为准。
- 邮件正文里的抬头字段，例如单位名称、税号、地址、电话、开户行、银行账户。

如果分组规则有歧义，先停下来确认。不要静默拆分整笔 `1800` 记录。

## 工作流

### 1. 解析来源

如果来源是飞书 wiki 链接：

1. 先运行 `lark-cli wiki spaces get_node --as user --params '{"token":"..."}'`。
2. 如果解析结果是 `docx`，继续用 `lark-cli docs +fetch --as user --doc '<url>'` 读取内容。
3. 找到用户指定的章节，例如 `可以报销（api）`。

如果章节里是 `<image token="...">` 图片而不是纯文本表格：

1. 用 `lark-cli docs +media-download --as user --token '<token>' --output '<name>' --overwrite` 下载每张图。
2. 放到一个专门的工作目录里。

如果来源是本地截图：

1. 直接读取截图。
2. 提取其中可见的日期、金额、交易单号。

### 2. 提取并核对记录

每条支付记录至少要提取：

- 日期和时间
- 展示金额
- 交易单号（如果能看到）
- 对应的附件文件名

如果记录是图片，做 OCR。优先使用本地 OCR，避免把敏感支付记录发到外部服务。在当前环境里，用 Swift 调 Apple Vision 是一个合适方案。

在起草邮件前，先核对：

- 记录条数
- 这些记录折算成多少个 `900` 单位
- 与用户目标总额是否一致

示例核对规则：

- `-899.96` -> 1 个 `900`
- `-1799.95` -> 1 个不可拆的 `1800`，不是 2 次独立申请

如果展示金额和按整数开票金额之间有尾差，例如 `11699.58` 与 `11700`，要明确说明这是显示尾差，不要当成异常。

### 3. 先给出分组方案

除非用户明确说“直接做”，否则先把分组方案列给用户确认。

当用户要求 `6` 封 `1800` 和 `1` 封 `900`，且 `1800` 记录不能拆分时：

- 每笔 `1799.xx` 记录单独成一封 `1800`
- 其余 `899.xx` 记录两两配成一封 `1800`
- 如果最后剩一笔 `899.xx`，则单独作为一封 `900`

分组列表要具体到日期和金额，方便用户快速复核。

### 4. 创建草稿

在用户明确确认发送前，只创建飞书邮箱草稿，不直接发送。

优先使用：

- `lark-cli mail +draft-create --as user ...`

附件路径必须用相对路径。先把附件统一重命名成可读文件名，再放到同一个工作目录，例如：

- `2025-12-29_225240_1799.98.jpg`

邮件主题要让用户在飞书邮箱里一眼能看懂。推荐格式：

- `YYMMDD-YYMMDD开票和明细请求（1800元）`
- `YYMMDD开票和明细请求（900元）`

### 5. 设置正文格式

正文以用户确认过的模板为准。一个常见模板是：

```html
<p>你好，</p>
<p>应学校财务报销要求，需要贵方开具<strong>发票以及细则单</strong>，因此麻烦你们提供。<br>本次报销对应的支付记录如附件所示。<br>本次开票总金额为1800元。</p>
<p>公司名称：中国科学技术大学<br>税号：12100000485001086E<br>单位地址：安徽省合肥市金寨路96号<br>电话号码：0551-63602363<br>开户银行：中行合肥南城支行<br>银行账户：184203468850<br>总开票金额：1800元</p>
```

只改动用户要求变更的字段，例如金额、单位信息或某些加粗样式。

如果草稿已经创建，只是想修改格式：

1. 先运行 `lark-cli mail +draft-edit --draft-id '<id>' --inspect`
2. 再运行 `lark-cli mail +draft-edit --draft-id '<id>' --patch-file './patch.json'`

这类开票邮件草稿优先用 `set_body`。保留已有附件。

## 串行写入规则

飞书邮箱草稿写入可能触发并发冲突，例如 `1236006`。

为了避免这个问题：

- 创建草稿时尽量串行执行
- 编辑草稿时串行执行
- 发送草稿时串行执行
- 如果用户重视可追踪性，每发出一封就立即查一次状态

除非用户明确接受失败后重试，否则不要并发批量写草稿。

## 发送规则

没有用户的明确确认，不要发送。

用户确认后：

1. 用 `lark-cli mail user_mailbox.drafts send --as user --params '{"user_mailbox_id":"me","draft_id":"..."}'` 逐封发送草稿。
2. 记录返回的 `message_id`。
3. 用 `lark-cli mail user_mailbox.messages send_status --as user --params '{"user_mailbox_id":"me","message_id":"..."}'` 查询投递状态。
4. 把每封邮件的结果清晰汇报给用户。

状态含义：

- `1`：正在投递
- `4`：投递成功
- `3`：退信
- `6`：审批拒绝

如果当前还是 `1`，要明确告诉用户“仍在投递中”，不要说成“已经成功送达”。

## 输出前检查清单

在交付草稿或执行发送前，确认：

- 每个附件都挂到了正确的邮件分组上
- 没有把 `1800` 记录静默拆开
- 每封邮件正文里的 `1800` 或 `900` 与它的附件总额对应
- 收件人和主题正确
- 如果用户要求 `发票以及细则单` 加粗，HTML 里仍然保留该格式
- 如果已经发送，投递状态也已经查过

## 最后怎么汇报

对用户的最终汇报保持简短、具体：

- 共创建了多少封草稿，或发送了多少封邮件
- 每封邮件的主题
- 每封邮件包含哪些日期和金额的记录
- 如果已经发送，当前投递状态是什么

只有状态查询明确返回已送达时，才说明“送达成功”。
